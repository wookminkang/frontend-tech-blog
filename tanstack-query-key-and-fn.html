TanStack Query에서 queryKey와 queryFn은 어떻게 동작하나요?

핵심 요약

queryKey는 캐시를 식별하는 고유 키입니다. 배열 형태로 선언하며, queryFn 내부에서 사용하는 모든 변수를 포함해야 합니다. queryFn은 데이터를 가져오는 비동기 함수로, 실패 시 반드시 에러를 throw해야 TanStack Query가 에러 상태를 인식합니다.

왜 이런 문제가 발생하나요?

TanStack Query는 queryKey를 기준으로 캐시를 저장하고 불러옵니다. 키가 같으면 같은 캐시를 사용하고, 키가 다르면 새로운 요청을 보냅니다.

queryKey 구조 예시
['posts']→// 전체 포스트
['posts', 1]→// id가 1인 포스트
['posts', { status: 'draft' }]→// draft 상태의 포스트
['posts', userId, { page: 2 }]→// userId가 바뀌면 새 요청
규칙 1

queryFn의 모든 의존성을 queryKey에 포함

외부 변수를 queryFn에서 참조하지만 queryKey에 빠뜨리면, 그 변수가 바뀌어도 캐시가 유효하다고 판단해 새 요청을 보내지 않습니다.

규칙 2

fetch 실패 시 에러를 직접 throw

fetch는 4xx, 5xx 응답에도 에러를 throw하지 않습니다. 응답 상태를 확인하고 실패 시 직접 throw해야 isError가 작동합니다.

규칙 3

queryKey는 직렬화 가능한 값만 포함

함수, 클래스 인스턴스, Symbol 등 직렬화가 안 되는 값은 queryKey에 사용할 수 없습니다. 문자열, 숫자, 객체, 배열만 사용하세요.

예시 코드 — 잘못된 사용

Bad — queryKey에 의존성 누락, fetch 에러 미처리
import { useQuery } from '@tanstack/react-query';

// queryFn에서 userId를 사용하지만 queryKey에 포함하지 않았습니다.
// userId가 바뀌어도 이전 사용자의 데이터가 계속 표시됩니다.
function useUserPosts(userId: number) {
  return useQuery({
    queryKey: ['posts'],  // userId 누락
    queryFn: () => fetchPostsByUser(userId),
  });
}

// fetch는 4xx/5xx에 에러를 throw하지 않습니다.
// isError가 false인 채로 에러 응답이 data에 담깁니다.
function usePost(id: number) {
  return useQuery({
    queryKey: ['posts', id],
    queryFn: async () => {
      const res = await fetch(`/api/posts/${id}`);
      return res.json(); // 404여도 에러 없이 진행됩니다.
    },
  });
}

올바른 사용법

Good — 의존성 포함, 에러 throw
import { useQuery } from '@tanstack/react-query';

// queryFn에서 사용하는 userId를 queryKey에 포함합니다.
// userId가 바뀌면 자동으로 새 요청이 발생합니다.
function useUserPosts(userId: number) {
  return useQuery({
    queryKey: ['posts', 'user', userId], // 의존성 포함
    queryFn: () => fetchPostsByUser(userId),
  });
}

// fetch 응답 상태를 확인하고 실패 시 에러를 throw합니다.
function usePost(id: number) {
  return useQuery({
    queryKey: ['posts', id],
    queryFn: async () => {
      const res = await fetch(`/api/posts/${id}`);
      if (!res.ok) {
        throw new Error(`포스트 조회 실패: ${res.status}`);
      }
      return res.json();
    },
  });
}
Good — 복합 파라미터와 staleTime 설정
// 쿼리 파라미터를 객체로 묶어 queryKey에서 관리합니다.
function usePosts({ page, status, userId }: PostsParams) {
  return useQuery({
    queryKey: ['posts', { page, status, userId }],
    queryFn: () => fetchPosts({ page, status, userId }),
    // 5분간 신선 상태를 유지합니다. 이 시간 안에는 재요청하지 않습니다.
    staleTime: 1000 * 60 * 5,
  });
}

// 컴포넌트에서 사용
function PostList({ userId }: { userId: number }) {
  const { data, isLoading, isError, error } = useUserPosts(userId);

  if (isLoading) return <p>불러오는 중...</p>;
  if (isError) return <p>{error.message}</p>;

  return (
    <ul>
      {data.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}

정리

규칙설명
queryKey는 배열로 선언계층 구조를 표현하고, 내용이 바뀌면 새 요청이 발생합니다
모든 의존성을 queryKey에 포함누락하면 캐시 무효화가 되지 않아 오래된 데이터가 표시됩니다
fetch 응답 상태 확인4xx/5xx 시 직접 throw해야 isError가 작동합니다
staleTime 설정자주 바뀌지 않는 데이터의 불필요한 재요청을 방지합니다
