왜 React에서 배열의 index를
key로 사용하면 안 되나요?

2026년 4월 14일 3분 읽기 React, key, 리스트 렌더링
React는 key를 기준으로 이전 렌더링과 현재 렌더링의 컴포넌트를 비교합니다. 배열의 index를 key로 사용하면 리스트 순서가 바뀔 때 React가 잘못된 컴포넌트를 재사용해 예상치 못한 버그가 발생합니다.
왜 이런 문제가 발생하나요?

React는 리스트를 렌더링할 때 각 아이템을 key 값으로 추적합니다. key가 동일하면 React는 "같은 컴포넌트"라고 판단하고 DOM을 재사용합니다.

index를 key로 사용하면 아이템을 추가하거나 삭제하거나 순서를 변경할 때, 실제로는 다른 아이템인데 key가 같아져 React가 잘못된 컴포넌트를 재사용하게 됩니다.

이로 인해 발생하는 문제입니다.

  • 입력창(input)의 값이 의도치 않은 아이템에 남아있음
  • 애니메이션이 잘못된 아이템에 적용됨
  • 체크박스 상태가 뒤바뀜
  • 불필요한 리렌더링으로 성능 저하
잘못된 사용 (Bad Example)
Bad tsx 
const TodoList = ({ todos }: { todos: Todo[] }) => {
  return (
    <ul>
      {todos.map((todo, index) => (
        <TodoItem key={index} todo={todo} />  // index를 key로 사용
      ))}
    </ul>
  );
};
todos 배열 맨 앞에 새 아이템을 추가하면 기존 아이템들의 index가 모두 1씩 밀립니다. React는 key=0인 컴포넌트가 여전히 존재한다고 판단해 첫 번째 TodoItem을 재사용합니다. 결과적으로 로컬 상태(input 값, 체크박스)가 엉뚱한 아이템에 붙게 됩니다.
올바른 사용법 (Good Example)
Good tsx 
const TodoList = ({ todos }: { todos: Todo[] }) => {
  return (
    <ul>
      {todos.map((todo) => (
        <TodoItem key={todo.id} todo={todo} />  // 고유 id 사용
      ))}
    </ul>
  );
};
각 아이템의 고유한 id를 key로 사용하면 배열 순서가 바뀌어도 React가 올바른 컴포넌트를 추적합니다.
고유 id가 없을 때 tsx 
// 데이터 생성 시점에 id 부여 (렌더링 중 생성 금지)
const newTodo = {
  id: crypto.randomUUID(),
  text: '새 할일',
  done: false,
};
정리

key는 React가 리스트 아이템을 구별하는 유일한 기준입니다. index는 배열 순서가 바뀌거나 아이템이 추가·삭제될 때마다 바뀌기 때문에 key로 적합하지 않습니다.

반드시 각 아이템의 고유한 id를 key로 사용하세요. 리스트가 정적이고 순서가 절대 바뀌지 않는 경우에만 index가 허용됩니다.
